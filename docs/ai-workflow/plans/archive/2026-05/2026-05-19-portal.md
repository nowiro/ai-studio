# Plan: portal — Native Federation full rollout + manifest discovery

## Goal

Move the remaining 9 apps (tire-shop, library, school-journal, bookstore, tools-shop, toy-shop, individual-wizard, business-wizard, tetris-game) behind the portal as lazy-loaded Native Federation remotes, and add a manifest-discovery endpoint so adding a new MFE is one config entry.

## Context

After audit Portal exists (Phase 3 of `portal-elements-keycloak.md`). The plan was `accepted` and ADR-0009 documented the choice. Web Component build-element targets exist for all 12 apps. Native Federation wired for a subset only.

## Scope

| In                                                                         | Out                                                   |
| -------------------------------------------------------------------------- | ----------------------------------------------------- |
| `federation.config.js` per remaining 9 apps                                | Cross-runtime sharing (Angular only)                  |
| Manifest discovery: `apps/portal/public/manifest.json` listing all remotes | Server-rendered shell                                 |
| Boot-time health check (skip a remote if 404)                              | Hot reload of federation manifests at runtime         |
| Lazy route registration in portal shell                                    | Auth federation (handled separately by Keycloak plan) |

## Risks & mitigations

- **Risk:** Bundle size grows due to duplicated singletons — Mitigation: explicit `shared: { '@angular/core': { singleton: true } }`; bundle analyser before/after.
- **Risk:** Federation runtime errors on missing remote — Mitigation: T004 health probe + graceful fallback to inline message in shell.

## Rollback

Per-remote opt-in. Manifest defaults to currently-federated apps. Adding `federation.config.js` does not break standalone builds; deleting it reverts to standalone-only.
